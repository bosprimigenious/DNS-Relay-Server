## 摘要

本报告围绕北京邮电大学计算机网络课程设计课题——DNS 中继服务器展开。我们在 Linux/WSL 环境下使用 C 语言实现了一个运行于 UDP 53 端口的 DNS 中继程序：根据本地配置文件 `dnsrelay.txt`，对域名查询执行**本地拦截**、**本地解析**或**上游转发**三种策略之一，并将结果返回客户端。

实现过程中，我们重点处理了 RFC 1035 报文格式、网络字节序转换、域名指针压缩、Transaction ID 映射、`select()` 事件驱动主循环等关键问题。测试表明，系统在本地解析、域名拦截、公网中继及 QTYPE 边界场景下均能给出符合协议预期的响应。本报告从需求背景、系统设计、实现思路、测试分析与总结反思五个层面，对项目全过程进行说明。

**关键词**：DNS · UDP · RFC 1035 · 中继服务器 · select · 网络字节序

## 第一章 需求分析

### 1.1 背景与课题意义

#### DNS 在互联网中的角色

当用户在浏览器地址栏输入 `www.bupt.edu.cn` 并按下回车时，计算机并不会直接连接这个字符串。操作系统会先把域名交给 DNS 解析服务，询问「这个域名对应的 IPv4 地址是什么？」；得到答案（例如 `123.127.134.10`）后，才会向该 IP 发起 HTTP/TCP 连接。

因此，DNS 处于应用层与传输层之间的关键位置：它负责把**人类可读的域名**映射为**机器可用的 IP 地址**。没有 DNS，互联网的大规模部署几乎不可能实现。

#### 本课题要做什么

本课程设计不要求我们实现一个功能完整的递归 DNS 服务器（不必维护根区、顶级域区、完整缓存体系等），而是实现一个**轻量级 DNS 中继（Relay）服务器**：

1. 在本机 **UDP 53 端口** 监听客户端 DNS 查询；
2. 查阅本地策略表，决定对该域名**拦截**、**本地应答**还是**转发上游**；
3. 构造或转发 DNS 响应，返回给发起查询的客户端。

这种「中继 + 本地策略」模型，与校园网、企业网中常见的 DNS 过滤、内网域名映射场景十分接近，具有明确的工程背景。

#### 与生产级 DNS 的差异（刻意简化）

| 能力 | 生产级递归 DNS | 本课题实现 |
|------|----------------|------------|
| 域名数据库 | 完整递归查询链 | 仅本地 txt 配置表 |
| 缓存 | TTL 感知缓存 | 无 |
| 传输协议 | UDP + TCP（大包） | 仅 UDP |
| 记录类型 | A/AAAA/MX/CNAME 等 | 主要处理 A 记录 |
| 并发模型 | 多进程/异步 I/O | 单线程 + select |

我们有意聚焦 RFC 1035 报文、UDP Socket、`select()` 多路复用和本地策略表，把课程知识点落到一份可编译、可演示、可测试的完整程序上。

### 1.2 一次 DNS 查询在本系统中的生命周期

理解「一条查询从进入到离开」的全过程，有助于把握整体设计：

1. **客户端**（浏览器、`nslookup` 等）向本机 53 端口发送 UDP 查询报文，报文内含域名、QTYPE（查询类型）和 Transaction ID；
2. **主循环**通过 `select()` 感知 socket 可读，调用 `recvfrom` 读入报文；
3. **协议模块**解析 Question 段，提取域名与 QTYPE；若报文非法，返回 FORMERR；
4. **配置模块**在 `dnsrelay.txt` 中查找域名：
   - 命中 `0.0.0.0` → 构造 NXDOMAIN 响应（拦截）；
   - 命中真实 IP 且 QTYPE=A → 构造 A 记录响应（本地解析）；
   - 命中真实 IP 但 QTYPE≠A → 构造空 NOERROR 响应；
   - 未命中 → 进入上游中继；
5. **中继模块**替换 Transaction ID，转发至 `114.114.114.114`，等待响应后还原 ID 并回传客户端；
6. 客户端收到响应，完成一次解析。

> **设计要点**：每一步都有明确的「失败路径」——格式错误、上游超时、表满等场景均不会 silent failure，而是返回对应 RCODE，保证客户端行为可预期。

### 1.3 功能需求

| 模式 | 触发条件 | 对外行为 | 典型场景 |
|------|----------|----------|----------|
| **本地拦截** | 配置 IP 为 `0.0.0.0` | RCODE=3（NXDOMAIN） | 屏蔽广告/恶意域名 |
| **本地解析** | 配置非零 IPv4 | 返回 A 记录（TTL=300s） | 内网域名、课程测试域 |
| **上游中继** | 域名不在配置表 | 转发上游，还原 ID 后返回 | 访问 baidu.com 等公网域 |

配置文件：`参考资料/dnsrelay.txt`，每行 `IP 域名`，支持 `#` 注释。启动时一次性加载；加载失败则退化为纯中继模式。

### 1.4 非功能需求

- **协议合规**：报文格式、字节序、QTYPE 一致性符合 RFC 1035；
- **CPU 友好**：主循环用 `select()` 带 10ms 超时，避免忙等待；
- **模块化**：协议、配置、ID 映射、主控分文件实现；
- **可测试性**：支持 `nslookup`/`dig`/自研脚本分别验证三种模式。

### 1.5 DNS 协议要点（与本项目相关）

DNS 报文经 **UDP** 传输，单条上限 **512 字节**，结构为：**12 字节首部** + Question/Answer/Authority/Additional 区段。

**首部关键字段：**

| 字段 | 宽度 | 作用 |
|------|------|------|
| ID | 16 bit | 匹配请求与响应；中继时必须正确还原 |
| FLAGS | 16 bit | QR（查询/响应）、RCODE（结果码）等 |
| QDCOUNT | 16 bit | Question 段记录数（本系统要求 =1） |
| ANCOUNT | 16 bit | Answer 段记录数 |

**域名编码**：长度前缀标签，如 `bupt` → `\x04bupt\x00`；应答可用 **指针压缩**（高 2 位 `11`）引用 Question 中已出现的域名。

**本系统使用的 RCODE：**

| 值 | 名称 | 含义 |
|----|------|------|
| 0 | NOERROR | 成功；非 A 本地命中时为空应答 |
| 1 | FORMERR | 报文无法解析 |
| 2 | SERVFAIL | 上游超时或转发失败 |
| 3 | NXDOMAIN | 本地拦截 |

## 第二章 系统设计

### 2.1 总体架构

采用「**单进程 + 单监听 Socket + 事件驱动主循环**」：

```
  客户端 ──UDP:53──► 主控（select 循环）
                         │
           ┌─────────────┼─────────────┐
           ▼             ▼             ▼
        协议模块      配置模块      中继模块
      解析/构造响应   dnsrelay.txt   114.114.114.114
```

**选型理由：**

- **单进程单线程**：DNS 查询报文短、逻辑简单，课程并发量下单线程足够；避免多线程带来的 ID 表竞态；
- **事件驱动**：`select()` 在无流量时不占满 CPU，有流量时及时响应；
- **模块分离**：协议细节与业务调度解耦，便于单独调试与替换（如将来升级为 epoll）。

### 2.2 模块划分

| 模块 | 文件 | 职责 |
|------|------|------|
| 协议层 | `dns_protocol.*` | 报文解析、域名编解码、各类响应构造 |
| 配置层 | `config.*` | 加载/查询本地域名—IP 表 |
| ID 映射层 | `id_map.*` | 记录中继时的 ID 与客户端地址，超时清理 |
| 主控层 | `main.c` | Socket 生命周期、select 循环、三路分支调度 |

模块间仅通过头文件声明的接口交互，主控层不直接操作报文字节偏移。

### 2.3 主循环与事件驱动

**问题**：若主循环在无数据时不断 `recvfrom`，会 busy loop，CPU 占用接近 100%。

**方案**：`select(sockfd, timeout=10ms)` —— 有数据立即处理；无数据最多等 10ms 再进入下一轮，兼顾响应延迟与 CPU 占用。

**分支逻辑**（文字描述）：

1. 报文长度 < 12 → 静默丢弃（可能是扫描流量）；
2. 解析失败 → FORMERR；
3. 配置命中 `0.0.0.0` → NXDOMAIN；
4. 配置命中非零 IP：QTYPE=A → 本地 A 记录；否则 → 空 NOERROR；
5. 未命中 → 上游中继；失败/超时 → SERVFAIL。

### 2.4 三种业务分支的设计思路

#### （1）本地拦截

配置 `0.0.0.0 域名` 表示「该域名不应被解析」。不转发上游，直接返回 NXDOMAIN。客户端（浏览器、`nslookup`）会理解为「域名不存在」，达到屏蔽效果。

#### （2）本地解析

配置真实 IP 时，本机扮演「该域名的应答者」。复制原查询的 Header + Question，在 Answer 段填入 A 记录，NAME 字段用指针压缩引用 Question 中的域名。

**重要边界**：仅 QTYPE=A 时返回 A 记录。对 MX/AAAA 等查询，即使域名在表中，也应返回空 NOERROR，否则 TYPE 与 QTYPE 不一致，部分客户端会异常。

#### （3）上游中继

未配置域名需转发公网 DNS。**Transaction ID 冲突**是中继的核心难点：多客户端可能使用相同 ID，转发时必须分配新 ID 并记录映射，收到上游响应后再还原 ID 回传。

当前为**同步中继**（发完等响应，超时 3 秒），处理一条中继时主循环暂时无法收新包；对课程规模可接受，高并发需改异步。

### 2.5 ID 映射表

固定容量 1024 条，环形槽位分配 + 线性探测找空位。每条记录含：原始 ID、新 ID、客户端 IP/端口、创建时间。

- 每次收包清理超过 **5 秒** 的过期记录；
- 表满时强制清空全部记录后重试一次，避免偶发高峰导致中继彻底失败。

## 第三章 关键实现说明

本章从「**遇到什么问题 → 为何这样设计 → 实际效果**」说明，不罗列 API。

### 3.1 网络字节序

**问题**：x86 为小端序，DNS 线缆格式为大端序；直接 memcpy 结构体会导致 FLAGS、COUNT 等字段颠倒，`dig`/`nslookup` 报格式错误。

**方案**：内存中用主机序操作，写入报文缓冲前统一 `htons`/`htonl`，读出后 `ntohs`/`ntohl`。FLAGS 位域与字节序交互复杂，封装专用转换函数屏蔽平台差异。

### 3.2 域名编解码与安全

**编码**：`bupt` → `\x04bupt\x00`；需校验标签长度 ≤63、总长 ≤255、无连续 `..`。

**解码**：顺序读标签；遇 `0xC0` 压缩指针则跳转继续读。

**风险与对策**：

| 风险 | 对策 |
|------|------|
| 循环指针导致死循环 | 跳转次数上限 10 |
| 指针偏移越界 | 每次访问前校验 offset < 512 |

### 3.3 配置表

启动时一次性加载，最多 4096 条。无效 IP 行跳过并告警，不影响其余记录。查表用大小写不敏感比较（DNS 域名语义不区分大小写）。

### 3.4 响应构造的统一原则

**保留原查询 Header + Question，只改响应相关字段**：

| 场景 | QR | RCODE | ANCOUNT |
|------|----|-------|---------|
| 拦截 | 1 | 3 | 0 |
| 本地 A | 1 | 0 | 1 |
| 格式错误 | 1 | 1 | 0 |
| 上游失败 | 1 | 2 | 0 |
| 非 A 本地命中 | 1 | 0 | 0 |

### 3.5 上游中继与超时

临时 UDP Socket + 3 秒 `SO_RCVTIMEO`。早期版本上游不可达时客户端 indefinite 等待；现改为返回 SERVFAIL，客户端可立即感知失败。

## 第四章 测试与结果分析

### 4.1 测试环境

| 项目 | 配置 |
|------|------|
| 操作系统 | WSL2 Ubuntu 24.04 |
| 编译器 | gcc 14，`-Wall -Wextra -std=c11` |
| 运行 | 默认 `sudo ./dnsrelay`；WSL 占用 53 时用 `DNS_RELAY_PORT=5353` |
| 工具 | `nslookup`、`dig`、`scripts/dns_query.py` |

选 WSL 因项目依赖 POSIX Socket API（`arpa/inet.h` 等），Windows 原生不提供。

### 4.2 测试用例

| 编号 | 命令 | 预期 | 验证点 |
|------|------|------|--------|
| 1 | `nslookup bupt 127.0.0.1` | `123.127.134.10` | 本地解析 |
| 2 | `nslookup sina 127.0.0.1` | `202.108.33.89` | 第二条本地记录 |
| 3 | `nslookup 008.cn 127.0.0.1` | NXDOMAIN | 拦截 |
| 4 | `nslookup baidu.com 127.0.0.1` | 公网 IP | 上游中继 |
| 5 | `nslookup -type=mx bupt 127.0.0.1` | 空 NOERROR | QTYPE 分支 |
| 6 | 双终端同时查 baidu.com | 均成功 | 基本并发 |

### 4.3 实测结果与分析

**环境**：`make` 零警告；测试端口 5353（正式验收用 53）。

**用例 1 — bupt**：`rcode=0, ancount=1, len=38`。命中配置 `123.127.134.10 bupt`，返回一条 A 记录，本地解析正确。

**用例 3 — 008.cn**：`rcode=3, ancount=0, len=24`。Answer 为空，NXDOMAIN 生效，拦截正确。

**用例 4 — baidu.com**：`rcode=0, ancount=4, len=91`。未命中本地表，经 114.114.114.114 中继，上游返回 4 条 A 记录；ID 还原正确。

**用例 5 — MX 查询 bupt**：NOERROR 且无 Answer，未误返 A 记录，QTYPE 分支正确。

完整输出见 `docs/test-output.txt`；PDF 附录建议附终端截图。

### 4.4 测试结论

三种核心模式均符合预期；边界场景（非 A 查询、上游超时、畸形报文）有明确错误响应，程序稳定不 crash。

## 第五章 总结

### 5.1 主要问题与解决

| 问题 | 现象 | 原因 | 解决 |
|------|------|------|------|
| FLAGS 解析异常 | dig 报格式错 | 位域与字节序 | 封装转换函数 |
| 指针死循环 | 畸形报文卡死 | 循环指针 | 跳转上限 10 |
| 指针越界 | 段错误 | 非法偏移 | offset 校验 |
| MX 误返 A | 类型不匹配 | 未判 QTYPE | 非 A 返回空 NOERROR |
| 上游挂起 | 客户端久等 | 无超时/无回包 | 3s 超时 + SERVFAIL |
| Windows 编译失败 | 缺头文件 | 非 POSIX | 改用 WSL2 |

这些问题多在**边界测试与工具联调**阶段暴露，说明网络程序不能只测 happy path。

### 5.2 收获

- 从概念到字节：亲手构造 DNS 报文，理解 RFC 1035 设计动机；
- 字节序是 Socket 编程的基本功；
- `select()` 事件驱动 vs 忙等待的直观对比；
- 模块化 C 工程与 Makefile 实践。

### 5.3 不足与改进

1. 同步中继阻塞主循环 → 可改 epoll 异步；
2. ID 映射表与同步模型未完全协同 → 异步化后表成为路由核心；
3. 无 TTL 缓存 → 重复查询每次都打上游；
4. 仅 IPv4 A 记录 → 可扩展 AAAA/CNAME/EDNS0。
